2. Interaction Types & Population Rules
Case
What it is
Each case thread contributes two row types:
• Case row — metadata (subject, open/close dates), the anchor row for the thread
• Case emails — all emails linked to the case, included for full LLM conversation context
Applied Filters
✓ Included when (any):
  • Case has ≥1 linked email with move_to_data_cloud__c = TRUE
  • Or is the parent case of a qualifying case
  • All emails in a triggered thread are included automatically
  • Case emails are incoming only (customer-sent; JFrog reply emails are excluded)
  • Linked account is present (accountid is not null)
✗ Excluded when (any field = TRUE):
exclude_from_bi__c internal_case__c is_test_account__c
Exception: Test account 001Tc00000XYSq5IAH is always included.
Thread / Session
Row type Thread Session
Case row Parent case ID  (own ID if no parent) Case's own ID
Case email Parent case ID  (own ID if no parent) Email's own ID
Edge Case: Multi-Level Case Hierarchies

Salesforce cases can form chains where a case is simultaneously a child of one parent and a parent to another — creating hierarchies deeper than 2 levels.

Rule: Pipeline supports a maximum depth of 2 levels. Cases at depth 3 or deeper — and all their descendants — are excluded entirely.

A case is excluded if its parent also has a parent (grandparent exists).

Scale: 516 cases in multi-level chains out of ~380,000 total. Chains reach up to 8 levels deep. The depth-2 cutoff handles this cleanly with no extra joins.

Example: A → B → C → D → E → F → G

Email
What it is
Emails not linked to any case (no parentid). Grouped into threads using Salesforce's email ThreadIdentifier field.

✓ Included when (all):

  • Email is incoming (sent by the customer, not JFrog)
  • Linked Task record has a valid accountid

4. Notes & Open Items
Thread/Session Rule
Every thread must have exactly one session row where Session ID = Thread ID. This is the anchor row: case metadata row for Cases, oldest email for Standalone Emails. Exception: for Gong, Thread (Salesforce ID) and Session (Gong ID) are always different — there is no anchor row where they match.
Data Lag: Missing Account ID
Salesforce sometimes populates account linkages asynchronously after a record is created. Records without a linked account at ingestion time are excluded and will be picked up automatically in a future run once the account is populated.
Resolved: Threads Missing Session Row

During testing, a small number of threads were identified where a session row with Session ID = Thread ID was missing.

Root cause: These were cases in multi-level hierarchies (depth 3+). The COALESCE(parentid, id) thread logic assigned them a thread ID belonging to a grandparent or higher ancestor — a case not included in the output as a metadata row, resulting in an "orphaned" thread.

Resolution: The depth-2 cutoff (see Section 3 Edge Case) excludes all cases at depth 3+ from the pipeline entirely, eliminating orphaned threads at the source.
